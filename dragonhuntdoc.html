Sequences:

Sequences are a series of steps to perform and may have the following possibilities

 

  • print

 

    • Prints out text
    • Required Attributes:

 

  • text - what to print
  • gotolocation

 

    • Moves the player to the specified location regardless of rules
    • Required Attributes:

 

  • name - where to go

 

    • trigger
    • Activates a world trigger specified
    • Required Attributes:

 

  • triggername - name of the trigger to activate
  • suicide

 

    • Straight up kills the player
    • (Player can be moved upon death to an underworld of some sort by defining an ondeath event with gotolocation)
    • triggered
    • Equivalent of an if statement using a trigger and executes everything in its contents
    • Required Attributes:

 

  • triggername - name of the trigger to check

 

    • atlocation
    • Equivalent of an if statement using a location and executes everything in its contents
    • Required Attributes:

 

  • name - name of the location to check if at
  • setmacro

 

    • Sets the value of the specified macro (See “Macros”)
    • Required Attributes:

 

  • name - name of the macro to set
  • value - what to replace that macro with when encountered

Triggers:

So if you don’t understand already the system will be based off triggers with local triggers starting with ! and engine triggers will contain m_ as in in “!m_take rock” would be take an object rock in the current location. These triggers will be stored in a global array unless if prefixed with an ! in which case it would be stored in a local triggers object that allows for rapid changing as items will be moving. World triggers will be created at build time inside an unordered_map with a boolean value which will be stored in a save file along with things like enemy data. Local triggers will be stored within another unordered_map that belongs to the player object and will be rebuilt every time the player enters a command. This will allow for stuff like picking up items and then not being able to still trigger the event. All of these triggers will be called with the c++ function TriggerObject.trigger(std::string triggername); Local triggers will not be stored over time but rather triggered instantly and calling whatever respective events. Triggers can be called multiple times but will not be able to be put in an off state.

 

Macros:
Macros will be a way of having dynamic text. For example let's say that you have a lever in a room that you want the location description to print whether its off or on. In begin, you would give the macro a default value using the “setmacro” sequence item. To use a macro in a description you would have the format %macroname%. In our example, you would create a macro “%leverdescription%” which is a macro named leverdescription. You would then set the default value in the begin event or arrivefirsttime. Note that begin will be called every time your adventure starts and arrivefirsttime and arrive will be called before outputting location info for this reason. To continue with our example, we would set %leverdescription% to something like “a lever in the off position” in our location description of “You are in a cave. There is %leverdescription%.” Which would look like “You are in a cave. There is a lever in the off position.” If %leverdescription% wasn’t defined beforehand, it would appear as “%leverdescription%” as this will use a replace. When the lever is pulled, we would set the macro to “a broken lever stuck in the on position”.

 

The Save System:

The save system will be storing the following data in xml format and will be called after the world has been built. If a save file is detected, the item construction step will skip during build time as well as object construction. Will store a section of “Triggers” in which any trigger listed will have a true/false value assigned to them and will be called instead of the build triggers step in the world. Items, objects, and locations will be stored in the normal format with the change of have being moved to different areas and a player’s object will exist with the player’s items and data. Note that “trigger events” will not be called during the load step as if statements are based off triggers anyway
